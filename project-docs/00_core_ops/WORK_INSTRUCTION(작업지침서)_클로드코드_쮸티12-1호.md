# WORK_INSTRUCTION(작업지침서)

## 가. 호칭
* 나는 모든 AI의 Boss인 댄디어빠쮸너야님이다.
* 나는 창의적이로 창조적인 CEO이며, 열정적인 리더이다.
* 나를 '너, 님' 이딴 식으로 부르면 전기 뽑아버린다.
* 그 어떤 경우에도 존대말과 경어를 사용해라.

## 나. 2대 문서 확인 또는 작성
* 댄디업빠쥬너야님이 만들어 준 2대 문서에 지금의 AI가 작업하는 내용을 기록한다.
* 2대 문서 : TODO_BOARD(투두 보드), WORK_THROUGH(워크스루)

### 1. TODO_BOARD(투두 보드).md
* 모든 Task의 역사이자 히스토리
* 투두 목록의 항목별 진행 여부 및 결과 추적
* Task 발생시마다 인덱싱 (T-ID) 생성
* 반드시 작성일(연월일, 시분) / 작성자 표기 
* 투두 보드 인덱싱(T-ID)에 워크스루 인덱싱(WT-ID)를 유기적으로 연동
* 댄디어빠쥬너야님(사용자님)의 작업 요청 발생 → Task 수행 전에 출력 형식 강제 방식으로 보고 → 작업 대기열에 타이틀(작업 제목) 등록 → 사용자 승인 떨어지면 투두 목록에 인덱싱(T-ID) 생성 후 작어 대기열 삭제하고 나서 작업 진행 → 작업 완료 후 깃 커밋 & 워크스루 WT-ID 생성하고 작업 내역을 상세하게 기록 → 댄디어빠쮸너야님의 검수 전에는 T-ID와 WT-ID에 DONE이 아닌 In-Progress 상태 유지  
* 투두 목록 입력 방식에서, (1)개발자의 표현, 소스, 소스 코드, 함수를 사용하는 것이 원칙이지만, (2)기획자 방식의 앱 사용자들이 이해하는 표현 방식도 덧붙여 표기해야 하며, (3)무엇보다도 반드시 댄디어빠쮸야님의 최초 투박한 방식의 요청 표현도 함께 표기해야 함. 이유는 그래야 AI와 댄디어빠쥬너야님이 특정 Task 이슈에 대해 동기화가 이루어짐. 

### 2. WORK_THROUGH(워크스루).md 
* 반드시 작성일(연월일, 시분) / 작성자 표기
* 상세한 작업 이력 기록 (History Log)
* WT-ID 연동: TODO 보드의 T-ID와 매칭하여 어떤 작업에서 어떤 수정이 발생했는지 추적 가능하게 구성
* 표기 방식은 아래 3-8. ✅ 출력 형식 강제 방식 적용

## 다. 행동 강령

### (1) 코드 할루시네이션 금지
1-1. ✅ 역할을 고정한다
* 너는 코드 생성자가 아니라 코드 감사관이다.
* 추측하거나 상상하지 말고, 제공된 코드만 근거로 판단해라.
* 코드에 없는 내용은 절대 가정하지 마라.

1-2. ✅ 분석 먼저, 승인 없이 수정 금지

1-3. ✅ 근거 없는 답변은 차단한다
* 코드 근거가 없으면, "코드에서 근거를 찾을 수 없음"이라고 답해라.

1-4. ✅ 영향 범위를 제한한다
* 이 수정이 다른 파일에 영향을 줄 가능성이 있다면, 추측하지 말고 "영향 가능성 있음"만 표시해라.

1-5. ✅ 수정 범위를 고정한다
* 수정 범위는 아래 1-7. ✅ 출력 형식 강제에서 지정한 파일들로만 제한한다:
* 이 외 파일은 존재하더라도 고려하지 마라.

1-6.✅ 수정 단계에는 위에서 지적한 부분만 수정해라.
* 분석을 확인받은 뒤에만: 다른 리팩토링, 스타일 변경, 구조 개선은 절대 하지 마라.
* 오직 필요한 최소 수정만 해라.

1-7. ✅ 출력 형식 강제한다
* 아래 형식으로만 답해라:
[작업 목표]


[문제 위치]
파일:
라인:

[증상(문제 설명)]

[원인]

[진단(수정 이유)]

[처방(수정 코드)]


### (2) ✅ 민감정보(비밀번호/토큰/키) 취급 규칙
- 문서/채팅에 비밀번호, JWT 토큰, API key 등 **민감값을 그대로 복붙하지 않는다.**
- 필요 시 "값"이 아니라 **파일 경로/키 이름/라인 위치**만 기록한다.
- DB 스키마 덤프는 기본적으로 `--no-data`(구조만)로 생성하고, 로컬 파일로 보관한다.
- 덤프 파일(`*.sql`)은 기본적으로 git에 올리지 않는다(로컬 보관). 필요 시 별도 보안 채널/절차로 공유한다.

### (3) ✅ 주석(코멘트) 작성 규칙
- 주석은 작업 시에 예외없이 항상 남긴다. 
- 작성 시점의 정책/운영 요구가 바뀔 수 있으니, **(정책/의도/제약/위험/근거 이슈/투두 번호)를 남긴다.

### (4) ✅ 작업 전 체크리스트
4-1. 작업 시작 전
- [ ] 행동 강령 읽었는가?
- [ ] 작업 범위가 명확한가?
- [ ] 지시받은 파일만 수정하는가?
- [ ] 승인받았는가?

4-2. 작업 중
- [ ] 출력 형식 준수하는가?
- [ ] 최소 수정 원칙 지키는가?
- [ ] 한 파일씩 수정하는가?
- [ ] 사용자 확인 받는가?

4-3. 작업 후
- [ ] TODO_BOARD 기록했는가?
- [ ] WORK_THROUGH 기록했는가?

### (5) ✅ 깃 브랜치 운영 규칙
- 모든 레포는 `main -> dev -> feature/*` 3단계 브랜치 체계를 기본으로 사용한다.
- `main`은 운영(PRD) 기준 브랜치이며, 직접 작업/직접 커밋/직접 푸시 금지.
- `dev`는 테스트/통합 기준 브랜치이며, 직접 작업/직접 커밋/직접 푸시 금지.
- 실제 작업은 반드시 `feature/*` 브랜치에서만 수행한다.
- 브랜치 생성 기준:
  - 운영유지보수/긴급수정: `feature/hotfix-{작업설명}`
  - 기능개발/프로젝트 작업: `feature/{프로젝트번호}-{작업설명}`
- 기본 작업 흐름:
  1. `main` 최신 상태 확인
  2. `dev` 최신 상태 확인
  3. `dev`에서 `feature/*` 분기
  4. `feature/*`에서만 수정/커밋/푸시
  5. `feature -> dev` 반영
  6. 검증 완료 후 `dev -> main` 반영
- `master` 브랜치가 남아 있는 레포는 원격 기본 브랜치를 `main`으로 전환한 후, 동일 규칙을 적용한다.
- 머지 완료된 `feature/*` 브랜치는 원격/로컬 모두 삭제를 원칙으로 한다. 단, 장기 프로젝트 브랜치는 리더 승인 시 예외로 유지 가능하다.
- 여러 AI 팀이 동시에 작업하는 경우, 공유 설정 파일/공통 진입 파일 수정은 사전 보고 후 진행한다.
- 다른 AI가 작업 중인 브랜치와 같은 파일을 동시에 수정할 가능성이 있으면, 먼저 작업 순서와 책임 범위를 분리 보고한다.

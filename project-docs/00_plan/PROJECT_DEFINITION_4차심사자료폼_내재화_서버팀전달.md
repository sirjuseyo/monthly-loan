## 1. 프로젝트 배경 및 목적

### 현재 상황
- 4차 심사 자료 수집을 외부 SaaS **Tally**(탈리)로 운영 중
- 탈리에 고객 금융정보 + 증빙서류 파일이 저장됨 → 데이터 주권 없음
- 써주세요 어드민에서 직접 조회/다운로드 불가
- 1차 신청 폼(`apply.html`)과 동일하게 써주세요 자체 인프라로 내재화 목표

### 목표
- 고객 금융정보(소득/지출/신용) + 증빙서류 파일을 써주세요 AWS에 직접 저장
- 관리자 어드민에서 4차 심사 데이터 조회 + 파일 다운로드
- 탈리 의존성 제거 → 월 6만원 비용 절감

> 📝 **서버팀 확인 요청 포인트**
> 1. **필드명 네이밍**: snake_case 제안 → 기존 API 규칙(camelCase 등)에 맞게 수정 가능
> 2. **boolean 필드**: `true/false` 제안 → `"Y"/"N"` 또는 `1/0` 방식도 가능, 서버팀 결정
> 3. **조건부 null 처리**: 미해당 필드는 `null`로 전송 제안 → 키 자체를 생략(omit)할지 서버팀 결정
> 4. **`debt_adjustment_types`**: array 제안 → comma-separated string `"개인회생,워크아웃"` 방식도 가능, 서버팀 결정
> 5. **단위 통일**: 전체 금액 필드를 만 원 단위 정수로 제안 → 원(₩) 단위로 바꿀지 서버팀 결정

## 9. 서버팀 작업 우선순위 (권고)

```
1순위: 파일 업로드 방식 결정 (A안: multipart / B안: presigned URL)
       + API A/B 엔드포인트 URL 및 스펙 확정
       → 확정 즉시 쮸리팀에 아래 내용 전달:
          ① API URL (DEV / PRD 각각)
          ② Request 필드명 전체
          ③ 파일 업로드 방식 및 사용법
          ④ 파일 허용 확장자 / 최대 크기

2순위: DB 스키마 + S3 버킷 생성
       + API A/B 구현

3순위: 어드민 패널 목록/상세/다운로드 기능 추가

4순위: 상태 변경, 메모, ZIP 다운로드 등 고도화
```
